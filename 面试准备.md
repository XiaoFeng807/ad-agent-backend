# 面试准备文档 — 广告投放助手项目

> 目标：让你面对面试官时，能说清楚每个模块为什么这么设计、解决了什么问题、有什么权衡取舍。

---

## 一、项目概览（30 秒自我介绍）

**一句话：**
> 一个基于 FastAPI + DeepSeek API 的多 Agent 广告投放管理平台，核心是 RAG 知识库 + Function Calling + 分层上下文管理。

**展开说（1 分钟）：**
> 这个项目主要解决广告投放人员的数据查询和分析需求。用户可以直接用自然语言问"今天ROAS多少"、"为什么下降了"，系统会自动拆解任务、查询数据库、检索知识库、最后给出带数据支撑的回答。核心架构是多 Agent 协同——一个 Orchestrator 调度三个子 Agent，分别负责数据查询、分析和知识检索。

---

## 二、架构总览（面试官必问）

### 分层架构

```
前端 (HTML + ECharts)
  ↓ HTTP/WebSocket
FastAPI 路由层
  ↓
Agent 层 (OrchestratorAgent + SubAgent)
  ↓
工具层 (tools.py) + RAG (ChromaDB) + 记忆层 (JSON/ChromaDB)
  ↓
LLM 层 (DeepSeek API)
```

**面试话术：**
> "我用了分层架构，路由层只做请求分发，业务逻辑全在 Agent 层。这样如果以后要把 FastAPI 换成别的框架，路由层改一下就行，Agent 层不用动。"

### 请求流程

```
用户: "为什么ROAS下降了"
  ↓
① 安全检测 → 敏感词/注入攻击检查
② 元认知预检查 → 问题是否在能力范围内
③ 记忆组装 → compose_context → L1滑动窗口 + L2短期 + L3画像 + L4业务
④ 任务拆解 → LLM 制定执行计划 [query_data → analyze_data]
⑤ ReAct 循环 → 调工具 → 拿结果 → 再调 → 最多5轮
⑥ 自我反思 → 检查回答质量，必要时修正
⑦ 质量评分 → 数据/完整/可读 三维度打分
⑧ 对话存储 → 存入 ChromaDB 对话记忆 + 重要性评分
  ↓
返回回答给用户
```

## 三、多 Agent 系统

### 架构

```
OrchestratorAgent (总指挥)
  ├── 数据助手 (SubAgent)  → query_data
  ├── 分析助手 (SubAgent)  → analyze_data  
  └── 知识助手 (SubAgent)  → query_knowledge_base
```

### Function Calling 原理

**面试话术：**
> "Function Calling 不是我自己写的函数调用框架，而是 LLM 原生能力。我把工具函数注册成 OpenAI 格式的 function schema，LLM 在生成回复时如果有需要，会返回 tool_calls 而不是直接回复文本。我收到 tool_calls 后执行对应的 Python 函数，再把结果塞回去让 LLM 继续生成。"

```
用户: "ROAS多少"
  ↓
Orchestrator 调用 LLM (with tools)
  ↓
LLM 返回: tool_call → query_data(query="ROAS")
  ↓
执行 query_data → 返回 {ROAS: 2.47}
  ↓
把结果注入上下文
  ↓
LLM 返回: "目前ROAS是2.47"
```

### ReAct 循环

```
Round 1: LLM 分析 "需要先查ROAS" → 调 query_data → 拿到 {ROAS: 2.47}
Round 2: LLM 分析 "ROAS偏低，需要查原因" → 调 analyze_data → 拿到 {原因: CPC上涨}
Round 3: LLM 有足够信息 → 直接回复用户
```

**上限 5 轮**，防止死循环。

### 竞争机制与结果融合

"如果一个问题有多角度答案（分析/对比/建议），我让 3 个 Agent 分别按保守/激进/详细三种风格生成答案，然后用 LLM 把三个答案的优点融成一个——保守的数据精度 + 激进的建议方向 + 详细的逐条展开。比单选好在哪里？单选丢掉了另外2个答案的信息量。"

## 四、RAG 知识库

### 检索流程

```
query: "ROAS怎么算"
  ↓
① 中文分词 (广告领域词典 + jieba)
② 向量检索 (ChromaDB + ChineseEmbedding)
③ 关键词检索 (BM25)
④ RRF 融合 (Reciprocal Rank Fusion)
⑤ 重排序 (ReRanker)
⑥ 输出 Top-K 结果给 LLM
```

### 关键技术点

**中文分词**：广告领域 80+ 专业词（ROAS/CPC/CTR/再营销等）+ jieba 回退

**Embedding**：自研 ChineseEmbedding（jieba 分词 + hash TF-IDF），不联网，中文适配比 all-MiniLM-L6-v2 好

**分块策略**：小-大分块。小块(200字)用来检索，大块(800字)作为 LLM 上下文。检索到小块后把对应的父块给 LLM，上下文更完整

**因果检索**：检测到"为什么/原因/导致"等信号词时，不走标准检索，而是拆成 原因分析 + 数据支撑 + 优化建议 三路分别检索，按因果重要性排序输出

### 数据清洗

每个文档入库前经过 3 道清洗：
1. HTML 标签过滤
2. 质量检查（长度/重复/噪声）
3. 交叉验证（多个来源不一致的标记为低置信度）

## 五、上下文管理（记忆系统）

### 四层架构

```
L1 滑动窗口: 保留最近 6 轮完整对话原文（按重要性保留，非时间）
L2 短期记忆: 30 分钟内对话摘要（JSON 文件）
L3 长期画像: 用户行为偏好 + 高频操作（30 天 TTL）
L4 业务记忆: 账户余额 + 决策记录 + 告警历史（ChromaDB）
```

### 重要性裁剪（替代纯滑动窗口）

"每个对话回合结束后，用规则打分（1-10分）。含数字/指标/操作的对话高分，'好的谢谢'低分。上下文满时不是删最旧的，是删最低分的。"

### 检索式记忆

"如果 LLM 发现自己上下文信息不够（用户问'之前那个计划'），它主动调用 search_conversation_memory 工具。和 RAG 的区别——RAG 检索行业知识，这个检索自己的历史对话，存在独立的 ChromaDB collection。"

### 时间衰减

"每次加载记忆时自动清理过期数据——短期 30 分钟、画像 30 天、业务 90 天。"

## 六、核心竞争力（面试亮点）

### 你能说的别人说不出的

| 亮点 | 怎么说 |
|------|--------|
| 因果 RAG | "普通 RAG 拆关键词，我拆因果关系——'为什么ROAS下降'会被展开成原因分析+数据+建议三路检索" |
| 重要性窗口 | "不和滑动窗口一样删最旧，删最不重要的——'ROAS下降原因'留着，'好的谢谢'删掉" |
| 结果融合 | "不单选一个答案，把保守的数据+激进的建议+详细的展开合并输出" |
| 中文 Embedding | "用的是 jieba + TF-IDF 自建 embedding，不是 all-MiniLM，中文更准" |

### 你踩过的坑

| 坑 | 怎么解决 |
|----|---------|
| 工具名不唯一导致 LLM 400 错误 | 所有函数名全局去重 |
| 循环导入 (circular import) | 按依赖层级拆文件，orchestrator 不导 agent_pool |
| BOM 导致 ast.parse 失败 | 统一 UTF-8 without BOM |
| ChromaDB 版本冲突 | 指定 embedding function 版本化，变更时自动重建 |

## 七、你能说出的不足（比硬吹加分）

### 面试官问"你这个项目有什么缺点"

**主动说出以下 3 点，比等面试官发现好：**

**1. 数据是 Mock 的**
> "目前数据是模拟生成的，没有对接真实的 Google Ads API。如果要上生产，第一件事就是接真实数据源。"

**2. Embedding 是规则法，不是模型**
> "我自建了中文 embedding 来避免联网依赖。但如果能下载模型，换成 bge-small-zh-v1.5 会更好，语义理解准确率预计提升 15-20%。"

**3. 元认知靠提示词，不是训练出来的**
> "目前的元认知是提示词级别的——告诉 LLM '不确定就去查'。如果有 GPU 和更多数据，我会用 LoRA 微调，让模型在训练中学会反思，效果会比提示词稳定得多。"

## 八、常见面试问答

**Q: 为什么用 ChromaDB 不用 FAISS？**
> "ChromaDB 内置了持久化、metadata 过滤和 embedding 函数管理，FAISS 需要自己搭全套。对于 demo 项目，ChromaDB 开发效率更高。如果数据量超过 100 万条，我会换 FAISS。"

**Q: 为什么不直接调 DeepSeek API 问所有问题？**
> "直接调 API 有两个问题：一是 LLM 没有实时数据（它只知道训练时的数据），二是 token 成本。所以我用 Function Calling 让 LLM 决定什么时候查数据库、什么时候检索知识库，只在需要推理和语言组织的时候才调 LLM。"

**Q: 多 Agent 和单 Agent 有什么区别？**
> "单 Agent 所有能力混在一起，提示词会非常长而且互相干扰。多 Agent 的好处是每个 Agent 提示词专注一件事，Orchestrator 做调度和汇总。代价是多了一次 Agent 间通信的延迟。"

**Q: 你的 RAG 和传统搜索有什么区别？**
> "传统搜索（Elasticsearch）基于关键词匹配，搜不到语义相近但字面不同的内容。RAG 用 embedding 做语义检索，'ROAS怎么算'和'广告回报率公式'会被映射到相近的向量空间。但 token 成本更高，所以我把 BM25 关键词检索作为补充——高精度召回+低精度语义召回融合。"
